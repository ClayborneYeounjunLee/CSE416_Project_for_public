# AI Travel Planner (Globird) — 프로젝트 구조 문서

> Stony Brook University **CSE416** 팀 프로젝트의 공개 복사본입니다.
> 목적지·기간·예산·동행·취향을 입력하면 **OpenAI(GPT-4o)** 가 하루 단위 여행 일정을 생성해 주고,
> 생성된 장소를 **Google 지도 경로 링크 / Booking.com 호텔 / Google Flights** 딥링크로 연결해 주는 **MERN + Firebase Hosting** 웹앱입니다.

> 이 문서는 기존 영어 [`README.md`](./README.md) 를 대체하지 않습니다. 코드를 직접 읽고 확인한 사실만 담은 한국어 상세 구조 설명서입니다.

- 🔗 **라이브 링크:** 없음 (별도 배포 URL 미확인 — Firebase Hosting 설정만 리포에 포함)
- 🏫 **분류:** 대학 팀 프로젝트 (coursework, 공개 복사본)

## ✨ 주요 기능 (사용자 관점)

- **Google 소셜 로그인** — Firebase Authentication + Google OAuth 팝업으로 로그인. 로그인 성공 시 사용자 정보(uid, email, 이름, 프로필 사진 등)를 백엔드(MongoDB)에 저장.
- **AI 여행 일정 생성** — 목적지 / 출발일 / 여행 일수 / 예산(Low·Medium·High) / 동행(Solo·Couple·Family·Friends) / 인원(성인·아동) / 관심 활동 / 추가 요구사항을 입력하면 OpenAI가 "Day별 장소" 형식의 일정을 만들어 줌.
- **일자별 경로 보기** — 생성된 일정에서 장소들을 추출해 **Google 지도 경로(출발지→경유지→도착지) 링크**를 하루 단위 버튼으로 제공.
- **호텔·항공 예약 딥링크** — 입력한 목적지·날짜·인원을 바탕으로 **Booking.com 검색 URL**과 **Google Flights 검색 URL** 을 자동 생성해 새 탭으로 연결.
- **장소 사진 미리보기** — 일정 속 장소를 클릭하면 **Unsplash** 이미지 API로 관련 사진 2장을 팝업으로 표시.
- **여행 계획 저장 / 조회 / 삭제** — 생성한 계획을 MongoDB에 저장하고, 프로필 페이지에서 내가 저장한 여행 목록을 보고 상세 열람·삭제.
- **프로필 관리** — 전화번호 / 국가 / 주소(2줄) / 표시 이름을 수정·저장.
- **(부분 구현) 피드백 게시판 / 문의 폼** — 피드백 작성·목록 조회, 이메일 문의 UI 컴포넌트가 존재하나, 현재 커밋된 서버에는 해당 API 라우트가 아직 구현되어 있지 않음(아래 "미완성/불일치" 참고).

## 🏗️ 시스템 구조

전형적인 **클라이언트–서버 분리형** 구조입니다. 단일 파일 앱이 아니라, React SPA(클라이언트)와 Express REST API(서버)가 나뉘어 있습니다.

```
[브라우저: React SPA]
   │  Google 로그인 (Firebase Auth, 클라이언트에서 직접)
   │  axios → http://localhost:8000/api/...
   ▼
[Express 서버 :8000]
   │  Mongoose
   ▼
[MongoDB]        [OpenAI API]  ← 서버가 /api/plan-trip 에서 호출
```

### 부팅 / 렌더 흐름 (클라이언트)
1. `src/index.js` — `ReactDOM.createRoot(...).render(<App/>)` 로 CRA 앱 마운트 (`<React.StrictMode>` 사용).
2. `src/App.js` — `<Router>` 안에서 라우트 정의. 앱 전역 상태는 `useState` 로 **`user`** 와 **`isLoginFinished`** 두 개만 최상위에서 관리(전용 상태관리 라이브러리 없음, prop drilling 방식).
3. 루트 경로 `"/"` 는 조건부 렌더:
   - `user` 없음 → `LoginPage`
   - 로그인은 됐지만 시작 전 → `FinishedLoginPage`(랜딩)
   - 시작 후 → `TravelPlan`
4. 로그인은 `firebase.js` 의 `signInWithGoogle()`(팝업) → 성공 시 `axios.post('/api/users')` 로 서버에 사용자 upsert → `FinishedLoginPage` 로 이동.

### 라우팅 (react-router-dom)
| 경로 | 컴포넌트 | 설명 |
|---|---|---|
| `/` | LoginPage / FinishedLoginPage / TravelPlan | 로그인·시작 여부에 따라 분기 |
| `/travel-plan` | TravelPlan | 여행 설문 입력 + AI 일정 생성 |
| `/profile` | Profile | 내 정보·저장한 여행 목록·상세 |
| `/finishedLoginPage` | FinishedLoginPage | 로그인 후 랜딩("Get Started") |
| `/profileedit` | ProfileEdit | 프로필 수정 폼 |

> 주의: `App.js` 는 `LoginPage`/`Profile`/`TravelPlan`/`FinishedLoginPage`/`ProfileEdit`/`FeedbackForm`/`FeedbackList` 를 import 합니다. 이 중 `FeedbackForm`·`FeedbackList` 는 import는 되어 있지만 어떤 `<Route>` 에도 연결되어 있지 않습니다. `Inquiry.js`·`SignInPage.js` 는 **어디에서도 import되지 않는 고아 파일**(파일만 존재)입니다.

### 핵심 함수 (프런트)
- `TravelPlan.handleGeneratePlan()` — `/api/plan-trip` 호출 → 응답 텍스트 파싱 → 경로/예약 URL 생성 → 결과 팝업.
- `extractLocations(planText)` / `parseItinerary(planText)` — OpenAI가 돌려준 **자연어 일정 텍스트를 정규식/문자열 split 으로 파싱**해 Day·장소 리스트로 변환.
- `createRouteUrl(locations, destination)` — Google 지도 경로 딥링크 조립.
- `fetchImagesFromUnsplash(query)` — Unsplash 검색 API 호출(아래 보안 주의).
- `handleSavePlan()` — `/api/save-travel-plan` 로 계획 저장.

### 핵심 라우트 (서버 `server.js`)
| 메서드 · 경로 | 기능 |
|---|---|
| `POST /api/users` | 사용자 upsert(없으면 생성). uid 기준 |
| `GET /api/users/:uid` | 사용자 조회 |
| `PUT /api/users/:uid` | 사용자 정보 수정 |
| `POST /api/plan-trip` | **OpenAI GPT-4o 호출** → 여행 일정 텍스트 + 호텔/항공 URL 반환 |
| `POST /api/save-travel-plan` | 여행 계획 저장 (URL 형식 검증 미들웨어 `validateUrls` 포함) |
| `GET /api/trips/:tripId` | 여행 계획 상세 조회 |
| `GET /api/users/:uid/trips` | 특정 사용자의 여행 목록 |
| `DELETE /api/trips/:tripId` | 여행 계획 삭제 |

`/api/plan-trip` 에서 GPT-4o 에 아래 형식으로 응답하도록 프롬프트를 고정합니다. 응답은 **JSON이 아니라 규격화된 자연어 텍스트**이며, 프런트가 이를 문자열 파싱합니다.

```text
- Day 1: 2024-12-01
  - Location : Central Park (park)
  - Location : Joe's Pizza (restaurant)
- Day 2: 2024-12-02
  - Location : ...
```

## 💾 저장소 / DB

### MongoDB (Mongoose) — 실제 영구 저장소
연결: `mongoose.connect(process.env.MONGODB_URI, ...)`. DB 이름/컬렉션은 Mongoose 모델명으로 자동 결정됩니다.

| 모델 (파일) | 컬렉션(자동) | 필드 | 비고 |
|---|---|---|---|
| `User` (`models/User.js`) | `users` | `uid`(unique·required), `email`(required), `displayName`(required), `phoneNumber`, `country`, `photoURL`, `address1`, `address2` + `timestamps`(createdAt/updatedAt) | Google 로그인 사용자 |
| `Feedback` (`models/Feedback.js`) | `feedbacks` | `userId`(required), `feedback`(required), `timestamp`(default now) | **모델은 있으나 서버에 라우트 미구현** |
| `TravelPlan` (`server.js` 내 인라인 정의) | `travelplans` | 위 §③ 스키마 참조 | 저장·조회·삭제 라우트 구현됨 |

> **게스트/오프라인 폴백:** 로그인 없이 계획을 저장하면 `TravelPlan.handleSavePlan()` 이 `user: "guest"` 로 저장합니다(`user && user.uid ? user.uid : "guest"`). 별도 오프라인/localStorage 캐시는 없습니다.

### localStorage
실질적으로 거의 사용하지 않습니다. 유일한 등장은 로그아웃 시 정리 코드 한 줄입니다.

| 키 | 용도 | 비고 |
|---|---|---|
| `user` | 로그아웃 시 `localStorage.removeItem('user')` 로 제거 | **어디에서도 `setItem('user', ...)` 로 쓰지는 않음** — 세션 상태는 React `useState` 에만 존재. 사실상 방어적/미사용 코드 |

### Firebase
- **Firebase는 인증(Authentication)과 호스팅(Hosting) 용도로만** 쓰입니다.
- **Firestore / Realtime Database는 앱 로직에서 사용하지 않습니다.** (`public/index.html` 은 Firebase CLI가 만든 기본 플레이스홀더로 각종 SDK를 나열하지만, CRA 빌드가 이 파일을 덮어쓰므로 실제 앱과 무관합니다.)
- Firebase 프로젝트 ID: **`cse416-7d42d`** (`.firebaserc`, `firebase.js`).

> Node.js 16+ / npm 8+ 권장 (기존 README 기준).

### 1) 서버 (`main/server`)
```bash
cd main/server
# 의존성은 상위 main/package.json 에 정의되어 있음

```
> 서버에는 `npm start` 스크립트가 없습니다. 직접 `node server.js` 로 실행합니다. `.env` 자동 로드를 위해 `node --env-file=.env server.js`(Node 20+) 또는 `dotenv` 사용을 권장합니다(현재 코드엔 dotenv import 없음).

### 2) 클라이언트 (`main/client`)
```bash

npm install
npm start              # CRA 개발 서버, 기본 http://localhost:3000
```
클라이언트는 API를 `http://localhost:8000` 으로 **하드코딩** 해 호출하므로, 서버를 먼저 켜두어야 로그인·일정 생성이 동작합니다.

### package.json 스크립트 요약
- **client** (`main/client/package.json`): `start`, `build`, `test`, `eject` — 표준 CRA.
- **server/root** (`main/package.json`): 실질 `test` 스크립트만 있고(플레이스홀더) start 스크립트 없음 → 서버는 `node server.js`.

## ⚠️ 미완성 / 코드-문서 불일치 (코드 기준 사실)

문서를 읽는 사람이 오해하지 않도록, 코드에서 확인된 불일치를 정리합니다.

- **피드백/문의 API 미구현:** `FeedbackForm`·`FeedbackList` 는 `/api/feedback` 을, `Inquiry` 는 `/api/send-email` 을 호출하지만, 커밋된 `server.js` 에는 이 라우트들이 **없습니다.** `Feedback` 모델과 `nodemailer` 의존성은 준비돼 있으나 연결 코드가 빠져 있습니다.
- **Skyscanner / Google Hotel API 미사용:** README/plan.txt의 언급과 달리 실제 호텔·항공은 **공개 검색 URL 딥링크**로만 구현됨.
- **`main/package.json` 의 잉여 의존성:** `firebase-admin`, `swagger-jsdoc/ui`, `react`(서버 폴더에는 불필요) 등은 현재 `server.js` 에서 쓰이지 않음.
- **하드코딩된 API 베이스 URL:** 클라이언트가 `http://localhost:8000` 을 직접 참조 → 배포 시 환경변수화 필요. (이 하드코딩 때문에 Firebase Hosting에 배포된 정적 프런트는 로컬 서버 없이는 로그인·일정 생성이 동작하지 않음)
- **`preferences` 무시:** `TravelPlan.handleGeneratePlan()` 이 `preferences` 를 요청에 담아 보내지만 서버 `/api/plan-trip` 는 이를 구조분해하지 않아 프롬프트에 들어가지 않음.
- **`options`(Halal/Vegetarian) 비활성:** 해당 체크박스 UI가 주석 처리돼 있어 항상 빈 배열로만 전송됨.
- **고아 컴포넌트:** `Inquiry.js`·`SignInPage.js` 는 어디에서도 import되지 않음. `FeedbackForm`·`FeedbackList` 는 import되지만 라우트에 연결 안 됨.
